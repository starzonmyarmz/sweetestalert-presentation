The Sweetest Alert

A <dialog> Migration Story

The Problem

SweetAlert has to go.

  • Bloated — includes too much for what we actually need
  • Requires overrides, extra boilerplate, and per-app customization
  • Inconsistent UX, versions, and implementations across PCO apps
  • Outdated — it doesn’t lean on modern, highly-supported features
  • Questionable — Openly promotes NSFW sponsors, Contains “protestware”

The Browser’s Gift

The browser has already solved this.

Focus trapping

Built-in — no library or manual event management needed.

Keyboard support

Escape to close, tab order — all handled natively.

::backdrop

Native overlay is baked in — no extra DOM required.

Top-layer promotion

Dialogs always render above everything. No z-index battles!

What Was Built

Sweetest Alert

  • Native <dialog> — way less code
  • Built for PCO — Tapestry tokens, our standards
  • Rails support via Stimulus + DOMPurify

The Strategy

Keep it simple. Ship it early.

The temptation
  • Big rewrite
  • Migration plan doc
  • All or nothing
Our approach
  • Inventory → POC
  • 2–3 instances at a time
  • Expand as we go
How It Happened

The process

  1. Inventory — every swal instance in Accounts
  2. Compare against the Figma spec
  3. POC in one low-traffic area accounts#4080
  4. Convert 2–3 at a time, growing in visibility and complexity
  5. Expand features and tests as each wave reveals new needs
Packaging

Extract. Document. Ship.

  • Extract to an NPM package with docs & demos
  • Switch Accounts over to the package
planningcenter.github.io/sweetest-alert
